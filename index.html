Linux Master Guide

1. Files & Directories (Navigation)

Terminal par apna raasta dhundna aur files manage karna.

Bash
# Main kahan hu? (Print Working Directory)
$ pwd
/var/www/html

# Is folder me kya hai? (Hidden files aur details ke sath)
$ ls -la

# Folder change karna
$ cd /etc/nginx    # Specific folder me jana
$ cd ~             # Home directory me wapas aana

# File copy, move (cut) aur rename karna
$ cp config.json config_bak.json   # Copy file
$ cp -r my_app/ backup/            # Folder copy (-r recursive)
$ mv old_name.txt new_name.txt     # Rename ya Move karna

# Delete karna (Danger: Recycle bin nahi hota!)
$ rm -rf folder_name/              # Forcefully pura folder delete

2. Permissions & Ownership

Security aur "Access Denied" errors fix karne ke liye.

chmod & chown
# chmod: Permissions badalna (Read=4, Write=2, Execute=1)
$ chmod 755 script.sh       # Owner ko sab allowed, baakiyo ko sirf padhna aur run karna
$ chmod +x script.sh        # Shortcut: Execute permission dena
$ chmod -R 755 /var/www     # Pura folder aur uski files ki permission set karna (-R)

# chown: Owner badalna (chown user:group filename)
[root@server]# chown mariya:developers index.html
[root@server]# chown -R www-data:www-data /var/www/html  # Web server ko access dena

3. Process & Resources (Task Manager)

ps, top, kill
# RAM aur CPU kitna use ho raha hai?
$ top          # Live monitor (press 'q' to exit)
$ htop         # Colorful & better (install karna padta hai)

# Running programs dekhna
$ ps aux | grep "node"    # Sirf node.js ke processes filter karke dikhao

# Program band (Kill) karna
$ kill 1234               # Normal kill (using Process ID)
$ kill -9 1234            # Force kill (Ziddi program ke liye)

4. Reading Logs & Searching Text

tail & grep
# Live logs padhna (Jaise hi error aayega screen pe dikhega)
$ tail -f /var/log/nginx/error.log

# File ke andar koi word dhundna (Grep)
$ grep "Exception" error.log        # Sirf wo lines layega jisme 'Exception' hai
$ grep -i "failed" mylog.txt        # -i for case-insensitive search

5. Systemd Services (systemctl)

Background processes aur services manage karne ke liye.

systemctl
$ systemctl status nginx    # Check if service is running
$ systemctl start nginx     # Start a service
$ systemctl stop nginx      # Stop a service
$ systemctl restart nginx   # Restart a service
$ systemctl enable nginx    # Start automatically on boot

6. Remote Access (SSH & SCP)

Remote servers connect karne aur files transfer karne ke liye.

ssh & scp
# SSH se connect karna
$ ssh root@192.168.1.10
$ ssh -i "mykey.pem" ubuntu@amazon-ip

# SCP se files bhejna (Local to Remote)
$ scp app.zip user@ip:/var/www/
$ scp -r build/ mariya@ip:/home/mariya/  # Folder transfer

7. Storage & Archiving (df, tar)

Disk & Tar
# Disk space check karna
$ df -h            # Hard drive space (Human readable)
$ du -sh /var/log  # Folder size check

# Tar se zip/unzip karna
$ tar -czvf backup.tar.gz /var/www/   # Create zip
$ tar -xzvf backup.tar.gz             # Extract zip

Git & GitHub Guide

1. Why we use Git? (Core Problem)

Bina Git ke, developers file ka naam project_final.zip, project_final_v2.zip rakhte the. Ek hi code par multiple log ek sath kaam nahi kar paate the, code overwrite ho jata tha. Git ne isey solve kiya "Version Control" aur "Branching" ke zariye.

Git Architecture (The 4 Stages)

💻 Laptop
Working Directory
Jahan aap code likhte ho (Untracked)
git add
📦 Index
Staging Area
Kise save karna hai, uska draft banta hai
git commit
💾 PC DB
Local Repo
Code ka ek version save ho gaya (Local)
git push
☁️ GitHub
Remote Repo
Cloud par code upload ho gaya (Team ke liye)

2. Basic Workflow & Branching

The Holy Trinity
# Naya project start karna
$ git init

# Code save karke GitHub bhejna
$ git add .
$ git commit -m "feat: login page"
$ git push origin main

# Team ka code lana
$ git pull origin main
Branches
# Main branch ko safe rakhne ke liye naya branch banana
$ git checkout -b "feature-payment"

# Branch me kaam khatam hone ke baad merge karna
$ git checkout main
$ git merge feature-payment

# Branch delete karna
$ git branch -d feature-payment

3. Advanced Git Concepts

Stash (Temporary Save)

Aap ek branch pe kaam kar rahe the, achanak boss ne kaha doosri branch pe urgent bug fix karo. Aapna adha adhura code commit nahi karna chahte. Wahan Stash kaam aata hai.

$ git stash          # Adha code temporary hide/save ho jayega, screen clear!
$ git checkout main  # Ab aaram se switch karo
$ git stash pop      # Wapas aane ke baad apna hide kiya hua code nikal lo

Rebase vs Merge

Dono code ko combine karte hain, par Rebase history ko ekdum seedhi (linear) line me rakhta hai bina extra "merge commit" banaye. Ye log (history) ko clean rakhta hai.

$ git pull --rebase origin main  # Apne commits ko main ke top par laga dega

Cherry-Pick

Puri branch merge nahi karni, sirf uss branch ka ek specific "Commit" chahiye.

$ git log --oneline              # Commit ki ID (hash) dekho, e.g. 5a3b2c1
$ git cherry-pick 5a3b2c1        # Uss ek commit ko apni current branch me apply kar lo

Docker & Containers

1. Why Docker? (VM vs Container)

Pehle "It works on my machine" ki problem bohot aati thi kyunki versions alag hote the. Uske baad log Virtual Machines (VM) use karne lage, par VM bohot RAM aur CPU khata tha (kyunki har VM me apna ek Pura OS hota tha). Docker ne isey solve kiya Containerization se, jo OS kernel ko share karta hai aur bohot lightweight hota hai.

Virtual Machine (Heavy)

App 1 + Bin/Libs
Guest OS (Ubuntu - 2GB RAM)
Hypervisor (VMWare/VirtualBox)
Host OS (Windows/Linux)
Hardware (CPU/RAM)

Docker Container (Lightweight)

App 1
App 2
Docker Engine (OS Shared)
Host OS (Windows/Linux)
Hardware (CPU/RAM)

2. Basic Lifecycle & Commands

Docker CLI
# 1. Blueprint download karo (Image)
$ docker pull nginx

# 2. Image ko zinda karo (Container banana)
# -d = detach (background me chalana), -p = Port forwarding (HostPort:ContainerPort)
$ docker run -d -p 8080:80 --name my_website nginx

# 3. Check karo kya chal raha hai
$ docker ps                 # Running containers
$ docker images             # Downloaded images ki list

# 4. Safai (Cleanup)
$ docker stop my_website
$ docker rm my_website      # Container delete
$ docker system prune -a    # Faltu ki saari images aur stopped containers uda do

3. Dockerfile (Custom Image Banana)

Apne khud ke code ki image banane ke liye hum project me ek `Dockerfile` rakhte hain.

Dockerfile
# Base image set karo
FROM node:18-alpine

# Container ke andar ek folder banao aur wahan jao
WORKDIR /app

# Dependencies ki file copy karke install karo
COPY package.json ./
RUN npm install

# Apne saare code ko copy karo
COPY . .

# Container run hone par ye command chalegi
CMD ["node", "app.js"]

4. Advanced Concepts (Volumes)

Docker Volumes (Data Persistence)

Container band ya delete hone par uske andar ka data (jaise Database records) udd jata hai. Data ko safe rakhne ke liye hum container ke folder ko apne laptop/server ke folder se map kar dete hain, jise Volume kehte hain.

# -v lagakar Host ka folder container ke folder se map kiya
$ docker run -d -p 3306:3306 -v /my/local/db_data:/var/lib/mysql mysql:latest

CI/CD Pipelines

1. Why CI/CD? (Automation Flow)

Pehle developer code likhta tha, usko .zip banata tha, FileZilla se server pe upload karta tha, aur manually command chalata tha. Ye bohot risky aur slow tha (Downtime aata tha).

CI/CD (Continuous Integration & Continuous Deployment) ek automated pipe hai. Developer GitHub pe push karta hai -> Server apne aap test run karta hai -> Code ko Build karta hai -> Aur automatically server par bina downtime ke deploy kar deta hai.

1. Plan & Code
2. Build (CI)
3. Test (CI)
4. Release & Deploy (CD)
5. Operate & Monitor

2. GitHub Actions (Real-World Workflow)

Ye file .github/workflows/deploy.yml mein rakhi jati hai. Jab bhi 'main' branch me code aaye, ye script chal jayegi.

deploy.yml
name: Production Deployment Pipeline

on:
  push:
    branches: [ "main" ] # Trigger point

jobs:
  build-and-deploy:
    runs-on: ubuntu-latest
    steps:
    - name: 1. Checkout Code
      uses: actions/checkout@v3

    - name: 2. Setup Node.js (Environment)
      uses: actions/setup-node@v3
      with:
        node-version: 18

    - name: 3. Install Packages & Run Tests (CI Phase)
      run: |
        npm install
        npm test

    - name: 4. Deploy to Server (CD Phase via SSH)
      uses: appleboy/ssh-action@master
      with:
        host: ${{ secrets.SERVER_IP }}         # Secure way: GitHub secrets use karna
        username: ${{ secrets.SERVER_USER }}
        key: ${{ secrets.SSH_PRIVATE_KEY }}
        script: |
          cd /var/www/myapp
          git pull origin main
          npm install
          pm2 restart all

3. Advanced Deployment Strategies

Blue-Green Deployment

Aapke paas do identical environments hote hain (Blue aur Green). Agar Blue live hai, to naya code Green me jayega. Test hone ke baad Load Balancer ka traffic instantly Blue se Green pe switch kar dete hain. Zero downtime, aur easily revert (rollback) kar sakte hain.

Canary Deployment

Naya code sirf 5% real users ko dikhaya jata hai (Load balancer ke zariye). Agar koi bug nahi aata aur system stable rehta hai, to dheere-dheere 20%, 50% aur 100% users ko naya code de diya jata hai.

Networking Deep Dive

1. Network Architecture (Mind Map)

DevOps engineers ko network ke har layer ka pata hona chahiye. Niche diya gaya tree structure basic se cloud network tak ka flow dikhata hai.

Networking Core

Fundamentals

IP Addresses (IPv4/IPv6), MAC Addresses, Subnetting, OSI Model

Protocols

TCP (Reliable), UDP (Fast), HTTP/HTTPS (Web), DNS (Resolution), SSH

Cloud / Advanced

VPC, Load Balancers, Reverse Proxy (Nginx), Firewalls (Security Groups)

2. IP Addressing & CIDR (Subnetting)

Cloud mein VPC (Virtual Private Cloud) banate waqt CIDR block samajhna bohot zaroori hai.

Public vs Private IP

  • Public IP: Internet par unique hoti hai. Aapke ghar ke router ya AWS EC2 ki ek public IP hoti hai.
  • Private IP: Local network (LAN/VPC) ke andar hoti hai (Jaise 192.168.x.x ya 10.0.x.x). Ye internet par direct access nahi hoti, secure rehti hai.

CIDR Block (e.g. /24)

CIDR batata hai ki ek network mein kitne IPs available hain.

  • /32 : Sirf 1 IP (Exact 1 machine ke liye rule).
  • /24 : 256 IPs milti hain (Chota network).
  • /16 : 65,536 IPs milti hain (Bada AWS VPC).

3. Proxies & Load Balancing

Forward Proxy vs Reverse Proxy

Forward Proxy (VPN jaisa): Client ke aage lagta hai. Client apni request proxy ko deta hai, aur proxy internet pe jata hai. Internet ko Client ki IP nahi dikhti (User identity hide karta hai).
Reverse Proxy (Nginx/HAProxy): Server ke aage lagta hai. Internet se aayi hui request pehle Reverse Proxy par aati hai, fir wo internally asal Backend (Node.js/C#) ko bhejta hai. (Server ki IP hide karta hai aur Security badhata hai).

Load Balancer (LB)

Jab bohot zyada traffic ho, toh ek server hang ho sakta hai. Isliye 4-5 servers banaye jate hain aur unke aage ek LB lagaya jata hai jo traffic ko sabme barabar (Round Robin) baant deta hai.

  • Layer 4 (Transport): Ye sirf IP aur Port (TCP/UDP) dekh kar traffic redirect karta hai. (Bohot fast hota hai).
  • Layer 7 (Application): Ye HTTP Headers, URL path (`/api` ya `/images`) padh kar smart routing karta hai. (Thoda slow par smart).

4. Essential Networking Commands

Troubleshooting ke waqt ye commands lifesaver hoti hain.

Network CLI
# 1. Connectivity Check Karna (Internet chal raha hai ya server zinda hai?)
$ ping google.com

# 2. Path tracing (Mera data server tak kin routers se hoke ja raha hai?)
$ traceroute google.com   # Linux me check karne ke liye

# 3. DNS Lookup (Domain ke peeche IP kya hai?)
$ nslookup mywebsite.com
$ dig mywebsite.com       # Detailed DNS records (A, MX, TXT) dekhne ke liye

# 4. Ports Checking (Kaunsa app konsa port use kar raha hai?)
$ netstat -tulpn | grep LISTEN
$ ss -tuln                # Modern fast alternative to netstat

# 5. Telnet/NC (Check karna ki kisi server ka port open hai ya firewall block kar rahi hai)
$ telnet 192.168.1.5 3306
$ nc -zv 192.168.1.5 3306

# 6. HTTP Requests (Terminal se API test karna)
$ curl -I https://api.myapp.com/users   # -I sirf headers aur Status code lata hai

DevOps Interview Q&A

Q1: Git Pull aur Git Fetch mein kya difference hai?

Answer:

  • Git Fetch: Ye sirf remote repository se latest changes ko download karta hai, par aapki local branch me merge nahi karta.
  • Git Pull: Ye pehle fetch karta hai aur phir automatically aapke current branch mein us code ko merge kar deta hai.

Q2: Docker Image aur Docker Container me kya fark hai?

Answer:

  • Image: Ye ek read-only template ya blueprint hota hai jisme aapka OS, code, libraries sab pack hoti hai.
  • Container: Jab aap ek Image ko Run karte ho, to wo ek active instance ban jata hai jisko Container kehte hain.

Q3: Hard Link aur Soft Link (Symlink) me kya difference hai (Linux)?

Answer:

  • Soft Link (Symlink): Windows ke "Shortcut" ki tarah hota hai. Ye sirf original file ka path (rasta) point karta hai. Agar original file delete ho jaye, to soft link broken ho jata hai.
  • Hard Link: Ye original file ke hard-disk memory address ko directly point karta hai. Agar original file delete bhi ho jaye, to Hard link wali file zinda rehti hai.

Q4: "Zombie Process" aur "Orphan Process" kya hota hai?

Answer:

  • Zombie Process: Process dead ho chuka hai, par parent process ne usko table se clear nahi kiya. Ye "Z" status me dikhta hai.
  • Orphan Process: Parent process crash ho gaya par child abhi tak chal raha hai. OS ka root process (PID 1) isko adopt kar leta hai.

Q5: Merge aur Rebase me kya difference hai (Git)?

Answer: Dono branch ko jodne ke kaam aate hain.

  • Merge: Ye dono branches ki history combine karke ek naya "merge commit" banata hai. History non-linear ho sakti hai.
  • Rebase: Ye aapke local commits ko temporary hata deta hai, main branch ke latest code ko lata hai, aur phir aapke commits ko uske aage set kar deta hai. Isse history bilkul straight (linear) dikhti hai.

Q6: Dockerfile me COPY aur ADD command me kya difference hai?

Answer:

  • COPY: Sirf normal files/folders ko local machine se container me copy karta hai. (Best practice).
  • ADD: Isme extra features hain. Ye URL se direct download kar sakta hai, aur .tar ya .zip file ko automatically extract (unzip) karke copy karta hai.
