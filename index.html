Linux Master Guide

1. Files & Directories (Navigation)

Terminal par apna raasta dhundna aur files manage karna.

Bash
# Main kahan hu? (Print Working Directory)
$ pwd
/var/www/html

# Is folder me kya hai? (Hidden files aur details ke sath)
$ ls -la

# Folder change karna
$ cd /etc/nginx    # Specific folder me jana
$ cd ..            # Ek step piche jana
$ cd ~             # Home directory me wapas aana

# File copy, move (cut) aur rename karna
$ cp config.json config_bak.json   # Copy file
$ cp -r my_app/ backup/            # Folder copy (-r recursive)
$ mv old_name.txt new_name.txt     # Rename ya Move karna

# Delete karna (Danger: Recycle bin nahi hota!)
$ rm -rf folder_name/              # Forcefully pura folder delete

2. Permissions & Ownership

Security aur "Access Denied" errors fix karne ke liye.

chmod & chown
# chmod: Permissions badalna (Read=4, Write=2, Execute=1)
$ chmod 755 script.sh       # Owner ko sab allowed, baakiyo ko sirf padhna aur run karna
$ chmod +x script.sh        # Shortcut: Execute permission dena
$ chmod -R 755 /var/www     # Pura folder aur uski files ki permission set karna (-R)

# chown: Owner badalna (chown user:group filename)
[root@server]# chown mariya:developers index.html
[root@server]# chown -R www-data:www-data /var/www/html  # Web server ko access dena

3. Process & Resources (Task Manager)

ps, top, kill
# RAM aur CPU kitna use ho raha hai?
$ top          # Live monitor (press 'q' to exit)
$ htop         # Colorful & better (install karna padta hai)

# Running programs dekhna
$ ps aux | grep "node"    # Sirf node.js ke processes filter karke dikhao

# Program band (Kill) karna
$ kill 1234               # Normal kill (using Process ID)
$ kill -9 1234            # Force kill (Ziddi program ke liye)

4. Reading Logs & Searching Text

tail & grep
# Live logs padhna (Jaise hi error aayega screen pe dikhega)
$ tail -f /var/log/nginx/error.log

# File ke andar koi word dhundna (Grep)
$ grep "Exception" error.log        # Sirf wo lines layega jisme 'Exception' hai
$ grep -i "failed" mylog.txt        # -i for case-insensitive search

5. Systemd Services (systemctl)

Background processes aur services manage karne ke liye.

systemctl
$ systemctl status nginx    # Check if service is running
$ systemctl start nginx     # Start a service
$ systemctl stop nginx      # Stop a service
$ systemctl restart nginx   # Restart a service
$ systemctl enable nginx    # Start automatically on boot

6. Remote Access (SSH & SCP)

Remote servers connect karne aur files transfer karne ke liye.

ssh & scp
# SSH se connect karna
$ ssh root@192.168.1.10
$ ssh -i "mykey.pem" ubuntu@amazon-ip

# SCP se files bhejna (Local to Remote)
$ scp app.zip user@ip:/var/www/
$ scp -r build/ mariya@ip:/home/mariya/  # Folder transfer

7. Storage & Archiving (df, tar)

Disk & Tar

# Tar se zip/unzip karna

Git & GitHub Guide

1. Why we use Git? (Core Problem)

Bina Git ke, developers file ka naam project_final.zip, project_final_v2.zip rakhte the. Ek hi code par multiple log ek sath kaam nahi kar paate the. Git ne isey solve kiya "Version Control" aur "Branching" ke zariye.

Git Architecture (The 4 Stages)

💻 Laptop
Working Directory
Jahan aap code likhte ho (Untracked)
git add
📦 Index
Staging Area
Kise save karna hai, uska draft banta hai
git commit
💾 PC DB
Local Repo
Code ka ek version save ho gaya (Local)
git push
☁️ GitHub
Remote Repo
Cloud par code upload ho gaya (Team ke liye)

2. Basic Workflow & Branching

The Holy Trinity
# Naya project start karna
$ git init

# Code save karke GitHub bhejna
$ git add .
$ git commit -m "feat: login page"
$ git push origin main

# Team ka code lana
$ git pull origin main
Branches
# Main branch ko safe rakhne ke liye naya branch banana
$ git checkout -b "feature-payment"

# Branch me kaam khatam hone ke baad merge karna
$ git checkout main
$ git merge feature-payment

# Branch delete karna
$ git branch -d feature-payment

3. Advanced Git Concepts

Stash (Temporary Save)

Aap ek branch pe kaam kar rahe the, achanak boss ne kaha doosri branch pe urgent bug fix karo. Aapna adha adhura code commit nahi karna chahte. Wahan Stash kaam aata hai.

$ git stash          # Adha code temporary hide/save ho jayega, screen clear!
$ git checkout main  # Ab aaram se switch karo
$ git stash pop      # Wapas aane ke baad apna hide kiya hua code nikal lo

Rebase vs Merge

Dono code ko combine karte hain, par Rebase history ko ekdum seedhi (linear) line me rakhta hai bina extra "merge commit" banaye. Ye log (history) ko clean rakhta hai.

$ git pull --rebase origin main  # Apne commits ko main ke top par laga dega

Cherry-Pick

Puri branch merge nahi karni, sirf uss branch ka ek specific "Commit" chahiye.

$ git log --oneline              # Commit ki ID (hash) dekho, e.g. 5a3b2c1
$ git cherry-pick 5a3b2c1        # Uss ek commit ko apni current branch me apply kar lo

Docker: A to Z Guide

1. Docker Architecture & Core Concepts

Docker ek client-server architecture use karta hai. Isme 3 main components hote hain: Client, Host (Daemon), aur Registry.

💻 Docker Client

CLI jahan hum commands chalate hain (e.g. docker run). Ye REST API ke through Docker Daemon se baat karta hai.

⚙️ Docker Host (Daemon)

Asli engine jo background me chalta hai (dockerd). Ye images banata hai, containers chalata hai, aur networking manage karta hai.

☁️ Registry (Docker Hub)

Jahan Images store hoti hain. GitHub for Docker. Hum yahan se image pull ya push karte hain.

2. Image vs Container (Difference)

📦 Docker Image

Read-only template. Blueprint jisme aapka OS, code, aur libraries pack hote hain. (Jaise Object Oriented Programming mein Class).

🏃‍♂️ Docker Container

Image ka running instance. Ye wo jagah hai jahan application actual mein execute hoti hai. (Jaise Class ka Object).

3. Essential Commands (Cheat Sheet)

Docker CLI
# --- IMAGES ---
$ docker pull nginx               # Image download karna
$ docker images                   # Downloaded images ki list
$ docker rmi nginx                # Image delete karna

# --- CONTAINERS ---
# Run karna (-d: background me, -p: port mapping, --name: custom naam)
$ docker run -d -p 8080:80 --name my_app nginx

$ docker ps                       # Sirf chalte hue containers dekhna
$ docker ps -a                    # Saare containers (band wale bhi) dekhna

$ docker stop my_app              # Container band karna
$ docker start my_app             # Wapas chalana
$ docker rm my_app                # Container delete karna (Pehle stop hona chahiye)

# --- TROUBLESHOOTING ---
$ docker logs my_app              # App ke errors/logs dekhna
$ docker exec -it my_app bash     # Chalte hue container ke andar ghusna (Terminal access)

# --- CLEANUP (DANGER) ---
$ docker system prune -a          # Saari unused images aur containers uday dega

4. Dockerfile Instructions (A-to-Z)

Custom image banane ke liye Dockerfile likha jata hai. Har instruction image mein ek nayi "Layer" add karta hai.

Instruction Description (Kaam kya hai?)
FROM ubuntu Base OS ya base image set karta hai. (Hamesha first line hoti hai).
WORKDIR /app Container ke andar folder banata hai aur baki commands wahi execute hoti hain.
COPY . . Aapke PC se files ko container me copy karta hai.
RUN apt update Build time par commands chalata hai (jaise software install karna).
ENV PORT=8080 Environment variables set karta hai.
EXPOSE 8080 Sirf documentation ke liye hai, batata hai ki app kis port pe chalegi.

Interview Q: CMD vs ENTRYPOINT

Dono container start hone par execute hote hain (Run time par). Par farq ye hai ki CMD ki command ko user `docker run` karte waqt override (change) kar sakta hai, jabki ENTRYPOINT fix rehta hai aur override nahi hota.

5. Storage & Networking

Storage (Data Persistence)

Container delete hone par uska data udd jata hai. Data bachane ke do tareeqe hain:

  • Bind Mounts: Aap apne PC ka ek specific folder (jaise /home/user/data) container ke folder se attach kar dete hain. (Local dev ke liye best).
  • Volumes: Docker khud ek folder banata hai aur usko manage karta hai. (Production / DBs ke liye best).
Volumes
$ docker run -v my_volume:/var/lib/mysql mysql    # Named Volume
$ docker run -v /home/user/data:/app/data nginx   # Bind Mount

Networking

Ek container dusre container se kaise baat karta hai?

  • Bridge (Default): Containers ek private network me hote hain, par ek dusre se IP ke through baat kar sakte hain.
  • Host: Container ka network hata diya jata hai, wo seedha server ka IP aur Port use karta hai.
  • None: Container ko internet access nahi hota (Strict security).
Network
$ docker network create my_network
$ docker run --network my_network my_app

CI/CD Pipelines

1. Why CI/CD? (Automation Flow)

Pehle developer code likhta tha, usko .zip banata tha, FileZilla se server pe upload karta tha, aur manually command chalata tha. Ye bohot risky aur slow tha.

CI/CD (Continuous Integration & Continuous Deployment) ek automated pipe hai. Developer GitHub pe push karta hai -> Server apne aap test run karta hai -> Code ko Build karta hai -> Aur automatically server par bina downtime ke deploy kar deta hai.

1. Plan & Code
2. Build (CI)
3. Test (CI)
4. Deploy (CD)

2. GitHub Actions (Real-World Workflow)

Ye file .github/workflows/deploy.yml mein rakhi jati hai. Jab bhi 'main' branch me code aaye, ye script chal jayegi.

deploy.yml
name: Production Deployment Pipeline

on:
  push:
    branches: [ "main" ] # Trigger point

jobs:
  build-and-deploy:
    runs-on: ubuntu-latest
    steps:
    - name: 1. Checkout Code
      uses: actions/checkout@v3

    - name: 2. Setup Node.js (Environment)
      uses: actions/setup-node@v3
      with:
        node-version: 18

    - name: 3. Install Packages & Run Tests (CI Phase)
      run: |
        npm install
        npm test

    - name: 4. Deploy to Server (CD Phase via SSH)
      uses: appleboy/ssh-action@master
      with:
        host: ${{ secrets.SERVER_IP }}         # Secure way: GitHub secrets use karna
        username: ${{ secrets.SERVER_USER }}
        key: ${{ secrets.SSH_PRIVATE_KEY }}
        script: |
          cd /var/www/myapp
          git pull origin main
          npm install
          pm2 restart all

3. Advanced Deployment Strategies

Blue-Green Deployment

Aapke paas do identical environments hote hain (Blue aur Green). Agar Blue live hai, to naya code Green me jayega. Test hone ke baad Load Balancer ka traffic instantly Blue se Green pe switch kar dete hain. Zero downtime, aur easily revert (rollback) kar sakte hain.

Canary Deployment

Naya code sirf 5% real users ko dikhaya jata hai (Load balancer ke zariye). Agar koi bug nahi aata aur system stable rehta hai, to dheere-dheere 20%, 50% aur 100% users ko naya code de diya jata hai.

Networking Deep Dive

1. Network Architecture (Mind Map)

DevOps engineers ko network ke har layer ka pata hona chahiye. Niche diya gaya tree structure basic se cloud network tak ka flow dikhata hai.

Networking Core

Fundamentals

IP Addresses (IPv4/IPv6), MAC Addresses, Subnetting, OSI Model

Protocols

TCP (3-Way Handshake), UDP, HTTP/HTTPS, DNS, SSH

Cloud / Advanced

VPC, Load Balancers, Reverse Proxy (Nginx), Firewalls (Security Groups)

2. OSI Model & TCP Handshake

Data ek system se dusre system tak physical cable se hote hue software tak kaise pohchta hai, ye OSI model define karta hai.

Layer Kaam (Function) Protocols & Devices
7. Application Network services provide karta hai user apps ke liye. HTTP, HTTPS, FTP, SMTP
6. Presentation Data formatting, Encryption (SSL/TLS) aur Compression. SSL, TLS, JPEG
5. Session Connections (sessions) ko establish, maintain, aur terminate karna. NetBIOS, PPTP
4. Transport End-to-End communication aur error recovery. Data ko "Segments" me todta hai. TCP, UDP
3. Network Best path (Routing) decide karna aur Logical Addressing (IP). Data yahan "Packets" banta hai. IP, ICMP, Routers
2. Data Link Physical addressing (MAC Address). Data yahan "Frames" banta hai. MAC, Switches
1. Physical Raw data (0s aur 1s) ko hardware cables/wireless ke through bhejna. Cables, Hubs, Fiber

TCP 3-Way Handshake (Interview V.Imp)

TCP reliable connection banata hai, uske liye pehle ye 3 steps hote hain:

  • 1. SYN: Client -> Server ("Bhai, sun raha hai?")
  • 2. SYN-ACK: Server -> Client ("Haan bhai, bol raha hu aur sun bhi raha hu.")
  • 3. ACK: Client -> Server ("Theek hai, main ab data bhej raha hu.")

3. IP Addressing & CIDR (Subnetting)

Cloud mein VPC (Virtual Private Cloud) banate waqt CIDR block samajhna bohot zaroori hai.

RFC 1918 (Private IP Ranges)

Ye IP address aapas mein (local network me) baat karne ke liye hote hain, internet pe nahi dikhte.

  • Class A: 10.0.0.0 - 10.255.255.255
  • Class B: 172.16.0.0 - 172.31.255.255
  • Class C: 192.168.0.0 - 192.168.255.255

CIDR Block (e.g. /24)

CIDR batata hai ki ek network mein kitne IPs available hain. IP 32 bits ki hoti hai. `/24` ka matlab pehle 24 bits network ke hain, aur baaki (32-24=8) hosts ke liye.

  • /32 : Sirf 1 IP (Exact 1 machine ke liye rule).
  • /24 : 256 IPs milti hain (Chota network).
  • /16 : 65,536 IPs milti hain (Bada AWS VPC).

4. DNS (Domain Name System) Deep Dive

Internet ka "Phonebook". Ye human-readable domain names ko IP addresses me convert karta hai. Port 53 use karta hai.

DNS Lifecycle (URL type karne par kya hota hai?)

  1. Browser Cache: Browser check karta hai ki kya pichle visit se IP yaad hai?
  2. OS Cache: Agar browser ko nahi pata, toh wo Operating System (/etc/hosts) se poochta hai.
  3. Resolver (ISP): OS request bhejta hai aapke internet provider ke DNS resolver ko.
  4. Root Server: Agar resolver ko nahi pata, toh wo global Root Server (.) ke paas jata hai.
  5. TLD Server: Root server usko Top Level Domain (jaise .com) server ke paas bhejta hai.
  6. Authoritative Server: TLD server final us server ka address deta hai jiske paas google.com ki exact IP (A Record) hai.

Important DNS Records

  • A Record: Domain name ko IPv4 address se map karta hai. (e.g. example.com -> 192.168.1.1)
  • AAAA Record: Domain name ko IPv6 address se map karta hai.
  • CNAME: Ek domain ko dusre domain (Alias) se map karta hai. (e.g. www.example.com -> example.com)
  • MX Record: Mail server batata hai. Emails kahan deliver hone hain.
  • TXT Record: Domain verification (Google Search Console) aur email security (SPF/DKIM) ke liye use hota hai.

5. Cloud Networking (VPC & Firewalls)

AWS ya Azure mein networking ka core setup samajhna.

☁️ VPC

Virtual Private Cloud. Aapka apna private network, ek digital building jisme bahar se koi bina permission nahi aa sakta.

🚪 IGW

Internet Gateway. Ye VPC ka main darwaza hai jisse bahar ka traffic andar aur andar ka bahar ja sakta hai.

🗺️ Route Table

Traffic ko raasta dikhata hai. Example: Agar 0.0.0.0/0 (Internet) pe jana hai toh IGW ki taraf jao.

🛡️ Sec. Groups

Instance-level Stateful Firewall. Ye decide karta hai ki EC2 server par konsa port (22, 80) allow hoga.

Public vs Private Subnet & NAT Gateway

Public Subnet mein wo servers (jaise Web Server / Load Balancer) rakhte hain jinko IGW se direct internet access chahiye.

Private Subnet mein secure chizein (jaise Database) rakhte hain jinhe koi internet se direct access na kar sake. Agar inko internet se updates download karne ho, to inhe ek NAT Gateway (jo Public subnet me hota hai) ke zariye bheja jata hai. NAT server ki private IP chupa kar public IP me badal deta hai.

6. Proxies & Load Balancing

Forward Proxy vs Reverse Proxy

Forward Proxy (VPN jaisa): Client ke aage lagta hai. Client apni request proxy ko deta hai, aur proxy internet pe jata hai. Internet ko Client ki IP nahi dikhti (User identity hide karta hai).
Reverse Proxy (Nginx/HAProxy): Server ke aage lagta hai. Internet se aayi hui request pehle Reverse Proxy par aati hai, fir wo internally asal Backend (Node.js/C#) ko bhejta hai. (Server ki IP hide karta hai, SSL termination aur Security badhata hai).

Load Balancer (LB)

Jab bohot zyada traffic ho, toh ek server hang ho sakta hai. Isliye 4-5 servers banaye jate hain aur unke aage ek LB lagaya jata hai jo traffic ko sabme barabar (Round Robin, Least Connections) baant deta hai.

  • Layer 4 (Network/Transport): Ye sirf IP aur Port (TCP/UDP) dekh kar traffic redirect karta hai. Pata nahi hota data ke andar kya hai. (Bohot fast hota hai).
  • Layer 7 (Application): Ye HTTP Headers, Cookies, URL path (`/api` ya `/images`) padh kar smart routing karta hai. (Thoda slow par smart).

7. Advanced Troubleshooting Toolkit

Firewall block kar rahi hai ya application down hai? Ye tools jawab denge.

Network CLI
# 1. ping - Check if server is alive (Uses ICMP protocol)
$ ping 8.8.8.8

# 2. traceroute - See all routers between you and target
$ traceroute google.com   # Linux me check karne ke liye

# 3. dig & nslookup - DNS query tool (Pata lagana A record ya CNAME kya hai)
$ dig google.com
$ nslookup example.com

# 4. ss / netstat - Check which ports are open/listening on your server
$ ss -tuln                # Shows TCP/UDP listening ports
$ netstat -tulpn | grep 80 # Kon port 80 use kar raha hai?

# 5. nc (Netcat) / telnet - Check if a specific port is reachable through firewall
$ nc -zv 192.168.1.10 3306  # Checks if Database port is open

# 6. curl - Make HTTP requests to test APIs
$ curl -v https://api.com   # Verbose mode (shows full request/response headers)

# 7. tcpdump - Packet Sniffer (Advanced: Capture raw network traffic)
$ tcpdump -i eth0 port 80   # See all HTTP traffic on interface eth0

Answer:

  • Git Pull: Ye pehle fetch karta hai aur phir automatically aapke current branch mein us code ko merge kar deta hai.
