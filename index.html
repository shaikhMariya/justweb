DevOps & Server Admin

Linux Important Notes
(Hinglish & Commands)

Deployments, Server Troubleshooting, File Permissions aur DevOps ke liye sabhi zaroori Linux commands with real examples.

1. Basic File & Directory Commands

Server par navigate karne aur files/folders manage karne ke liye basic terminal commands aana sabse zyada zaroori hai.

Terminal (Bash)
# 1. Main kahan hu? (Print Working Directory)
$ pwd
/var/www/html

# 2. Is folder me kya kya hai? (List files with details and hidden files)
$ ls -la

# 3. Folder change karna (Change Directory)
$ cd /etc/nginx    # Specific folder me jana
$ cd ..            # Ek step piche jana
$ cd ~             # Home directory me jana

# 4. Naya folder banana
$ mkdir my_project
$ mkdir -p /var/log/myapp/errors  # -p se parent folders bhi sath ban jate hain agar exist nahi karte

# 5. File copy aur move (cut) karna
$ cp config.json config_backup.json      # File copy
$ cp -r my_folder/ backup_folder/        # Pura folder copy (-r for recursive)
$ mv old_name.txt new_name.txt           # Rename karna
$ mv my_app.zip /var/www/                # Move karna

# 6. Delete karna (CAREFUL: Recycle bin nahi hota idhar!)
$ rm test.txt                            # File delete
$ rm -rf old_folder/                     # Pura folder forcefully delete (-r recursive, -f force)

2. File Permissions & Ownership

Linux mein security sabse badi cheez hai. Har file ke permissions 3 hisso me hote hain: User (u), Group (g), Others (o). Inme Read (r=4), Write (w=2), Execute (x=1) ka access diya jata hai.

chmod (Change Mode/Permissions)

  • 777: Sabko sab allowed (DANGER!) [4+2+1, 4+2+1, 4+2+1]
  • 755: Owner can do anything. Others can only read & execute. (Directories ke liye normal)
  • 644: Owner can read/write. Others can only read. (Files ke liye normal)

chown (Change Owner)

Jab aap root user se koi file banate ho to dusra normal user ya (www-data / nginx) usko access nahi kar pata. Wahan chown kaam aata hai.

Permissions Examples
# Permission badalna (Read, Write, Execute dena)
$ chmod 755 script.sh       # Ab is file ko koi bhi execute (run) kar sakta hai
$ chmod +x script.sh        # Shortcut: Sabko execute permission de do
$ chmod -R 755 /var/www     # -R (Recursive) pure folder aur uske andar ki saari files ke liye

# Owner badalna (chown user:group filename)
[root@server]# chown zayan:developers index.html
[root@server]# chown -R www-data:www-data /var/www/html/laravel_app  # Web server ko permission dena (Very Common in Deployments)

3. Process & Resource Management

Server slow kyu chal raha hai? Kaunsa app RAM kha raha hai? Koi atka hua program band karna ho? Ye sab yahan hoga.

Process Monitoring
# 1. Real-time CPU/RAM usage dekhna (Windows Task Manager jaisa)
$ top          # Basic built-in tool (Press 'q' to quit)
$ htop         # Colorful aur interactive tool (Install karna padta hai: apt install htop)

# 2. Running processes ki list nikalna
$ ps aux       # Saare processes detailed view me
$ ps -ef | grep dotnet  # Sirf '.net' wale processes ko dhundna (Filter karna)

# 3. Kisi atke huye process ko band (kill) karna
$ kill 1234    # Gently kill karna (PID 1234)
$ kill -9 1234 # Forcefully kill karna (Jab normal kill se kaam na ho)
$ killall node # Naam se kill karna (Saare node.js processes uday dega)

4. Search, Grep & Log Reading

Ek DevOps engineer ka aadha din log files padhne mein jata hai. Files ko edit kiye bina CLI par padhne ke tools aane chahiye.

Logs dekhna (cat, tail, less)

$ cat error.log        # Puri file ka text ek sath print kar dega
$ less error.log       # Badi file ko page-by-page padhne ke liye (Arrow keys work karengi, 'q' se quit)

# Sabse Important: Live Logs (Real time me naye errors dekhna)
$ tail -n 50 error.log # Last ki 50 lines dikhayega
$ tail -f error.log    # '-f' (follow) live output dikhata rahega (Ctrl+C to stop)

Find & Grep (Dhundna)

# FIND: File ka naam dhundna
$ find /var/www -name "config.js"   # /var/www folder me config.js kahan hai dhundo

# GREP: File ke ANDAR ka text dhundna
$ grep "Exception" error.log        # error.log me jahan bhi "Exception" likha hai wo line print karo
$ grep -i "failed" syslog           # -i case-insensitive search ke liye (Failed, failed dono aayega)
$ grep -r "DB_PASSWORD" /etc/       # /etc folder ki har file me DB_PASSWORD dhundo

5. Networking & Ports Checking

App start nahi ho rahi? "Port already in use" ka error aaya hai? Server ka internet check karna hai? Use these:

Network Commands
# 1. API ya URL se data / files download karna
$ curl http://localhost:5000/api/health   # CLI se API hit karne ke liye (Postman jaisa)
$ curl -I https://google.com              # Sirf HTTP Headers aur Status Code (200 OK) check karne ke liye
$ wget https://example.com/file.zip       # Direct file download karne ke liye

# 2. Port Checking (Mera app kis port pe chal raha hai?)
$ netstat -tulpn | grep LISTEN    # Saare open/listening ports dikhayega (Older systems)
$ ss -tuln                        # Modern fast alternative to netstat
$ lsof -i :8080                   # Port 8080 kon use kar raha hai? (App ka naam aur PID dega)

# 3. Connection check karna
$ ping google.com                 # Internet check karne ke liye
$ telnet 192.168.1.5 3306         # Database (3306 port) is server se access ho raha hai ya firewall block kar rahi hai?

6. Systemd Services (systemctl)

Jab aap production server par .NET ya Node app chalate ho, to aap chahte ho ki server restart hone par app khud start ho jaye, aur agar crash ho to automatically restart ho. Iske liye hum us app ki ek Service (Daemon) banate hain.

Service Management
# Nginx (Web Server) ya kisi custom app ko manage karna:

$ systemctl status nginx    # Service abhi chal rahi hai (Active) ya band hai (Dead)?
$ systemctl start nginx     # Service start karna
$ systemctl stop nginx      # Service band karna
$ systemctl restart nginx   # Restart karna (Downtime hoga)
$ systemctl reload nginx    # Bina downtime ke naya config apply karna (Best for Nginx!)

# Server boot par auto-start enable karna:
$ systemctl enable nginx    # Server restart hone pe Nginx khud start hoga
$ systemctl disable nginx   # Auto-start band karna

# Custom app ke logs dekhna (systemd wale apps ke liye)
$ journalctl -u kestrel-myapp.service -f   # .NET service ke live logs dekhne ke liye

7. Remote Access (SSH & SCP)

SSH (Secure Shell)

Apne laptop (Windows/Mac) se AWS/Azure ke cloud server ka terminal access karna.

# Password se login
$ ssh root@192.168.1.10

# Private key (Pem/Key file) se login
$ ssh -i "mykey.pem" ubuntu@amazon-ip

SCP (Secure Copy)

Apne laptop se files/builds seedha server par securely bhejna bina FTP clients ke.

# Local file ko server par bhejna
$ scp app.zip user@ip:/var/www/

# Pura folder bhejna (-r)
$ scp -r build/ user@ip:/home/user/

8. Storage & Tar Archives

Server ki memory full hone wali hai? Backup compress karna hai?

Disk & Zip Commands
# 1. Disk Space Check karna
$ df -h            # Puri hard drive ka space dikhayega (Human readable - GB/MB me)
$ du -sh /var/log  # Specific folder kitna size le raha hai check karna

# 2. Tar (Tape Archive) - Linux ka default zipping tool
# Create (Compress) - Folder ko tar.gz me badalna
$ tar -czvf myapp_backup.tar.gz /var/www/myapp/   # (c=create, z=zip(gzip), v=verbose, f=file)

# Extract (Unzip) - Tar file ko kholna
$ tar -xzvf myapp_backup.tar.gz                   # (x=extract)

# Normal Unzip
$ unzip package.zip -d /target/folder/

DevOps / Linux Interview Q&A

Q1: Hard Link aur Soft Link (Symlink) me kya difference hai?

Answer:

  • Soft Link (Symlink): Windows ke "Shortcut" ki tarah hota hai. Ye sirf original file ka path (rasta) point karta hai. Agar original file delete ho jaye, to soft link bekar (broken) ho jata hai. (Create: ln -s original.txt link.txt)
  • Hard Link: Ye original file ke hard-disk memory address (inode) ko directly point karta hai. Agar original file delete bhi ho jaye, to Hard link wali file zinda rehti hai aur data safe rehta hai. (Create: ln original.txt link.txt)

Q2: "Zombie Process" aur "Orphan Process" kya hota hai?

Answer:

  • Zombie Process: Ye wo process hai jo apna kaam complete karke band (dead) ho chuka hai, par iska Parent process abhi tak OS ko bata nahi paya hai ki child dead hai. Ye process table me "Z" ya "defunct" status se dikhta hai. (Ise directly kill nahi kar sakte, parent ko kill karna padta hai).
  • Orphan Process: Agar Parent process pehle hi kill/crash ho jaye, par uska child process abhi tak background me chal raha ho, to child anath (Orphan) ho jata hai. OS ka "init/systemd" (PID 1) process isko adopt kar leta hai.

Q3: chmod 777 ka matlab kya hai?

Answer: chmod se file permissions change hoti hai. 777 ek octal representation hai.

  • 4 = Read (r), 2 = Write (w), 1 = Execute (x). Inka total = 7.
  • Pehla '7' User (Owner) ke liye hai (All access).
  • Dusra '7' Group ke liye hai.
  • Teesra '7' Others (Duniya bhar ka koi bhi user) ke liye hai.

Ye permission security ke liye bohot khatarnaak hai, production me kabhi use nahi karna chahiye.

Q4: Cron Jobs kya hote hain?

Answer: Linux mein task scheduling ke liye cron ka use hota hai. Agar humein koi script (jaise DB Backup ya email sending) roz raat 12 baje automatically chalani hai, to hum crontab file me entry add karte hain. (Command: crontab -e).

# Format: Minute Hour Day Month DayOfWeek Command
0 0 * * * /path/to/backup_script.sh

Q5: Ek file me text search karke usko count kaise karein?

Answer: Grep ko pipe (|) aur word count (wc) ke sath combine karke.

$ grep "ERROR" mylog.txt | wc -l
# Ya direct grep se:
$ grep -c "ERROR" mylog.txt
